# 手書き end-to-end 求解ベンチ（実問題・実機手書き）

- 日付: 2026-06-13
- 問い: **実際にある定番の方程式を iPad に手書きで連続入力**したとき、認識→求解の通しでどれだけ正解できるか
- 入力: 中沢さん本人が iPad Safari の webapp に16問を順に手書き
- 処理: Mac サーバが OCR（**qwen2.5-VL 7B / ローカル Ollama**）→ `mathai` で求解、`web/captures/log.jsonl` に記録
- 採点: `spikes/solve/grade_session.py`（recognize→直後solveをペア化・重複除去・OCR一致は SymPy で意味的に判定）
- 問題と正解: `spikes/solve/problems.json`（期待解は SymPy で代入検証済、`build_problems.py`）

## 実験設計（フェアにするための前処理）

- 問題は一次/二次/分数/三次/根号/混合の **16 問**を厳選（手書きしやすい1行式）。
- **エンジン単体は事前に 16/16 正答**（クリーンなテキスト入力時）と検証済 → 本番の失敗は **OCR 由来**と切り分けられる。
- 確認カードは**あえて修正せず**「解く」を押す運用（生の OCR 精度を測るため）。

## 結果

| 指標 | 結果 |
|---|---|
| **OCR 一致（意味的）** | **15/16 (94%)** |
| **end-to-end 求解正答** | **15/16 (94%)** |

- 分数（`\frac{x}{2}+\frac{x}{3}=5`）、三次（`x^{3}-6x^{2}+11x-6=0`）、根号（`\sqrt{x+1}=2`）も**正しく認識・求解**。
- 唯一の失敗 **Q4**: 手書き `x²+2x-8=0` の**係数 2 を OCR が「1」と誤読**（`x²+1x-8=0`）→ 無理数解。数式自体は正しく解けており、誤りは**1桁の読み違い**。

## 解釈（正直に）

- ✅ **実問題でも end-to-end 94%**。ローカル完結（Mac/Ollama+SymPy）で、手書きの代数方程式を通しで解けることを実データで確認。
- 🔑 **唯一の失敗は確認カードで救える種類**。`x²+1x-8` は確認カードに出た時点で人が直せる（今回は計測のため直さず通した）。**確認カード＝OCR の安全網、検算＝求解の安全網**で役割が違う。
- ⚠️ **検算（verified）は OCR 誤りを検出できない**。Q4 は「誤読した式」としては正しく解け検算も通る。だから OCR の桁誤りは確認カードで止めるのが正解、という二層設計の根拠になった。
- ⚠️ 既知の弱点 = **係数など1桁の読み違い**（2↔1 等）。今後はプロンプト改善・確認カードの差分強調で軽減する。

## 計測上の学び（methodology）

- 初回の採点で「解く」の**二度押し**による重複ログが出現順対応をズラし、誤った 56%/75% を出した → **recognize→直後solveのペア化＋重複畳み込み**に修正。
- OCR 一致を latex 文字列の素一致で測ると `\frac{x}{2}` と `x/2` が不一致になる → **SymPy で同値判定**に修正（表記差を吸収、意味だけ見る）。
- 教訓: 自動採点は「重複・表記ゆれ・順序ズレ」を吸収する設計でないと数字を誤る。

## 再現方法

```bash
.venv/bin/python spikes/solve/build_problems.py          # 問題セット生成（正解 SymPy 検証）
.venv/bin/python spikes/solve/grade_session.py --mark    # 手書き開始前に基準記録
# iPad で http://<MacのLAN>:5173 を開き 16 問を順に手書き（認識する→解く）
.venv/bin/python spikes/solve/grade_session.py           # 採点
```

## 限界・次

- 16 問・1 セッション・1 筆者・OCR は Mac の 7B。**オンデバイス 3B での同ベンチ**は別途（速度・精度のトレードオフ確認）。
- 問題数を増やし（型別カバレッジ）、確認カードの**差分ハイライト**で OCR 桁誤りの取りこぼしを減らす。
- 連立・不等式など**エンジン未対応の型**を入れて対応範囲の地図を広げる（現状は単一変数方程式）。
