# 信頼性の定量化 — 各安全網は OCR 誤りを何%止めるか（研究テーマD）

- 日付: 2026-06-13
- 問い: OCR が間違えたとき、どの層が止めるのか。**確認カード**と**検算**の寄与を数値化する
- 手法: 正解16問（`spikes/solve/problems.json`）に**現実的な OCR 誤りを1文字だけ注入**し、`mathai` に通して4分類
- 実装: `spikes/solve/reliability_injection.py`

## 注入する OCR 誤り（実観測に基づく）

- **数字の見間違い**（手書きで紛れる対: 2↔1, 3↔8, 5↔6, 6↔0, 9↔4 …）。**Q4 の 2→1 もこの型**。
- **符号反転**（+ ↔ −）
- **指数ズレ**（x^2 → x^3 等）

## 4分類

| 分類 | 意味 | 気づける？ |
|---|---|---|
| engine_caught | アプリが「解けません」（unsupported） | ✅ 表示で気づく |
| verify_caught | 検算 verified=False | ✅ 検算が検知 |
| harmless | verified=True かつ解が元と一致（誤読が数学を変えず） | ― |
| **SILENT** | **verified=True かつ解が相違（Q4型）** | ❌ **確認カードだけ** |

## 結果（141 件の注入）

| 分類 | 件数 | 割合 |
|---|---|---|
| engine_caught | 1 | 1% |
| verify_caught | 8 | 6% |
| harmless | 0 | 0% |
| **SILENT** | **132** | **94%** |

- **機械の網（エンジン＋検算）が捕捉: 9/141 = 6%**
- **確認カードだけが救える（SILENT）: 132/141 = 94%**

## 解釈（重要）

- 🔑 **検算（代入検証）は OCR 誤りをほぼ救えない（6%）**。誤読された式も「その式としては正しく解ける」ので verified=True になる。検算は**求解の正しさ**を保証する層であって、**読み取りの正しさ**は保証しない。
- 🔑 **安全性の 94% を確認カード（人の目）が担う**。Q4 の失敗は偶然ではなく**構造的**だった。
- → 二層設計の役割が数値で確定: **確認カード＝OCR の網（主役）／検算＝求解の網（別目的）**。
- verify_caught の少数（6%）は、**求解自体が代入を満たさない狭いケース**（根号の無縁解・実数解なし等）で、OCR 誤り検出が目的ではない副作用的検知。

## この結果が示す次の一手

- 確認カードが安全性の主役なのに、現状は**人が気づくか頼み**。→ **確認カードを賢くする**価値が最大（不確実トークンの強調・複数回認識のクロスチェック・解の破綻を OCR に還流）。
  ※ 研究テーマ A は今回未選択だが、本データは A の**最強の根拠**。B/C と並行で「確認カードの差分強調」だけでも入れる価値が高い。

## 正直な限界

- 注入は**単一文字・16問**の模擬誤り（実 OCR 誤り分布そのものではない）。傾向は明確だが、実分布での再測は今後。
- 「確認カードが実際に何%の SILENT を人が捕まえるか」は人間試験が要る（本ベンチは"確認カードだけが救える領域の大きさ"を測ったもの）。
