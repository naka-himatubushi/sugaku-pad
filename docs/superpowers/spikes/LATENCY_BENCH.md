# 推論レイテンシ・ベンチ（MLX vs Ollama）

- 日付: 2026-06-13
- 目的: ローカル手書き OCR を**速くしてクラウドに負けない**ための実測
- 環境: MacBook（M5 Pro / 48GB）、モデル: qwen2.5-VL 7B
- 計測: `spikes/ocr/bench_mlx.py`（MLX）／warm 化済み web サーバ（Ollama）、実手書き `2x+19=9`

## 結果

| バックエンド | warm latency | 認識 | 備考 |
|---|---|---|---|
| **MLX (mlx-vlm, 4bit)** | **0.29s** | `2x + 19 = 9` ✓ | 起動時の初回ロード ~116s（一度きり） |
| Ollama (warm) | ~1.8s | `2x + 19 = 9` ✓ | 起動時ウォームアップで常駐済み |

- **MLX が約 6 倍速い**（同モデル・同認識）。
- **0.29s はクラウド API の往復（ネット遅延込みで通常 1〜数秒）より速い** → 「速度でクラウドに負けない」を満たす。

## 注記（正直に）

- 計測は 1 サンプル（指標値）。傾向は明確だが、複数問・複数字種での平均は今後拡張。
- MLX の弱点は**初回ロード ~116s**（グラフコンパイル含む）。実運用は **mlx-vlm の常駐サーバ or FastAPI プロセス内ロード**でロードを一度きりにし、以降 0.29s を享受する。
- Ollama も warm 化で初回 9s→常時 1.8s に改善済み（[server.py]）。手軽さは Ollama、最速はローカル MLX。

## 推奨

- **最速・完全ローカルを狙うなら OCR バックエンドを MLX へ**（起動時ロードは常駐で吸収）。
- 手軽さ重視や Studio 併用時は Ollama 継続でも可（warm 化済みで 1.8s）。
