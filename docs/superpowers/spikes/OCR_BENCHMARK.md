# 手書き数式 OCR 精度ベンチ（ローカル LLM）

- 日付: 2026-06-13
- 目的: ローカル LLM の手書き数式認識が**実際どれだけ通用するか**を数値化する
- データ: 中沢さん本人が webapp で実際に書いた手書き 5 問（`spikes/ocr/handwriting_eval/`）
- 手段: `spikes/ocr/eval_handwriting.py`（OCR は **qwen2.5-VL 7B / ローカル Ollama**）
- 指標: ①文字一致（正規化後）②end-to-end 求解一致（mathai で解いて同じ答えか）

## 結果

| 問題 | 種類 | 文字一致 | 求解一致 |
|---|---|---|---|
| 2x + 7 = 29 | 一次 | ✓ | ✓ |
| 2x + 19 = 9 | 一次 | ✓ | ✓ |
| 2/(x²−4) − 1/(x²+2x) | 分数式 | ✓ | ✓ |
| 2x² − 3x − 5 | 二次式 | ✓ | ✓ |
| 2x + 19 = 28 | 一次 | ✓ | ✓ |

- **文字一致: 5/5（100%）**
- **end-to-end 求解一致: 5/5（100%）**

## 解釈（正直に）

- 完全ローカル（Mac/Ollama）で、実手書きの代数・分数式を**初期セットで 100%** 認識・求解できた。
  対照: 印刷用 OCR の pix2tex は同種の手書きで**文字化け**（[SPIKE_RESULTS.md](SPIKE_RESULTS.md)）→ VLM の優位が明確。
- ⚠️ **サンプルは 5 件と小規模**。三角・指数・複雑な根号・乱雑な字などで崩れる可能性は残る。
  → だからこそ **確認カード（人が修正）＋ 検算レイヤー（代入で正しさ保証）** で“最後の数%”を担保する設計。
- 拡張: `handwriting_eval/` に画像と正解を足せば再計測できる。崩れる例を集めてプロンプト/前処理を改善する研究に使える。

## 面談での語り方

「ローカル LLM の手書き認識を**自分の実データで定量評価**（初期 5/5）。印刷用 OCR では破綻することも実測。完璧でない前提で**確認＋検算の二層**を入れて実用化した」= 計測に基づく設計判断。
