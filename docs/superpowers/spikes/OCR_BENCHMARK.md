# 手書き数式 OCR 精度ベンチ（ローカル LLM）

- 日付: 2026-06-13
- 目的: ローカル LLM の手書き数式認識が**実際どれだけ通用するか**を数値化する
- データ: 中沢さん本人が webapp で実際に書いた手書き 5 問（`spikes/ocr/handwriting_eval/`）
- 手段: `spikes/ocr/eval_handwriting.py`（OCR は **qwen2.5-VL 7B / ローカル Ollama**）
- 指標: ①文字一致（正規化後）②end-to-end 求解一致（mathai で解いて同じ答えか）

## 結果

| 問題 | 種類 | 文字一致 | 求解一致 |
|---|---|---|---|
| 2x + 7 = 29 | 一次 | ✓ | ✓ |
| 2x + 19 = 9 | 一次 | ✓ | ✓ |
| 2/(x²−4) − 1/(x²+2x) | 分数式 | ✓ | ✓ |
| 2x² − 3x − 5 | 二次式 | ✓ | ✓ |
| 2x + 19 = 28 | 一次 | ✓ | ✓ |

- **文字一致: 5/5（100%）**
- **end-to-end 求解一致: 5/5（100%）**

## 解釈（正直に）

- 完全ローカル（Mac/Ollama）で、実手書きの代数・分数式を**初期セットで 100%** 認識・求解できた。
  対照: 印刷用 OCR の pix2tex は同種の手書きで**文字化け**（[SPIKE_RESULTS.md](SPIKE_RESULTS.md)）→ VLM の優位が明確。
- ⚠️ **サンプルは 5 件と小規模**。三角・指数・複雑な根号・乱雑な字などで崩れる可能性は残る。
  → だからこそ **確認カード（人が修正）＋ 検算レイヤー（代入で正しさ保証）** で“最後の数%”を担保する設計。
- 拡張: `handwriting_eval/` に画像と正解を足せば再計測できる。崩れる例を集めてプロンプト/前処理を改善する研究に使える。

## オンデバイス候補モデル比較（2026-06-13 追記）

- 問い: **iPad（Apple Silicon, 8–16GB RAM）に載るサイズ**のモデルで手書き精度はどこまで出るか
- 候補: 端末内に現実的なサイズの vision LLM 3 種を同じ手書き 5 問で横並び
- 速度: 同一画像（`h2.png`）を 2 回叩いた warm latency（モデル常駐, ローカル Ollama Q4_K_M）

| モデル | params | 実体(Q4) | 文字一致 | end-to-end 求解 | warm latency | 8GB iPad 適合 |
|---|---|---|---|---|---|---|
| **Qwen2.5-VL 3B** | 3.8B | **3.2GB** | 5/5 | **5/5** | **0.23s** | ✅ 余裕 |
| Gemma 4 E2B | 5.1B | 7.2GB | 5/5 | 5/5 | 2.98s | △（後述） |
| Qwen2.5-VL 7B | 7B | 6.0GB | 5/5 | 5/5 | 0.26s | ✗（16GB 機向け） |

→ **端末内 OCR の第一候補 = Qwen2.5-VL 3B**。精度は 3 種とも実データ 5/5 で並んだが、**最小（3.2GB）・最速（0.23s）**で 8GB iPad に最も無理なく載る。

### Gemma 4 E2B の評価（正直に）

- 精度は 7B/3B と同等（5/5）で、**LaTeX 出力が一番きれい**（空白ノイズなし）だった。
- ただしローカル Ollama 実測では **約 13 倍遅く（2.98s vs 0.23s）、実体も倍以上（7.2GB / 生 5.1B params）**。
- "E2B"=実効 2B は **Per-Layer Embeddings (PLE) で実効 RAM を ~2–3GB に抑える設計**だが、それを活かすには PLE 対応ランタイムが要る。素の 4bit ロードだと 5.1B 相当＋vision tower でメモリ・速度とも 3B に劣る。
- 結論: **MLX/iOS 上で footprint と速度を確認できるまで第一候補にはしない**。精度マージンが欲しい 16GB 機の比較対象として保留。

### パーサ堅牢化（この計測で発見・修正）

- qwen3b は `\frac { 2 } { x ^ { 2 } - 4 }` のように**空白・入れ子波括弧**付き LaTeX を出し、当初 e2e 4/5 だった（認識は正しいのにパーサが分母の入れ子 `{2}` で詰まった）。
- `mathai/latex_input.py` で **`^{2}`→`^(2)` の括弧化を分数展開より前**に実行するよう変更 → 入れ子が消えて **e2e 5/5 に回復**（全 29 テスト緑）。OCR 文字列の表記ゆれを計算層が吸収する好例。

### 機種マッピング（8–16GB 前提）

- **8GB（iPad 無印 / Air 下位 / mini）→ Qwen2.5-VL 3B**（~3GB、0.23s、5/5）。
- **16GB（iPad Pro M 系上位）→ 3B で余裕、精度マージンが欲しければ 7B も選択可**。
- いずれも MLX Swift（`VLMRegistry.qwen2_5VL3BInstruct4Bit`）で端末内実行可（[ON_DEVICE_FEASIBILITY.md](ON_DEVICE_FEASIBILITY.md)）。

## iPad 実機 実測（2026-06-13 追記）— オンデバイスの本命検証

- 端末: **iPad Pro 13" M4（iPad16,5, 8GB モデル）**
- 実行: `ios/` アプリ `Spike D: MLX ベンチ`（MLXVLM, `Qwen2.5-VL 3B 4bit`）を実機で 2 回計測
- 計測値は os_log の `MLX-BENCH-RESULT` 行を Mac から `idevicesyslog` で自動回収

| 指標 | 実測（2回平均） |
|---|---|
| モデルロード（キャッシュ後） | 約 7.0s |
| cold latency | 約 2.5s |
| **warm latency** | **約 2.38s** |
| 生成スループット | 約 55 tok/s |
| **RAM ピーク** | **約 3.2GB**（ロード後 ~3.1GB） |
| 残メモリ余裕 | 約 1.96GB |
| 認識結果 | `2x+19=9` ✓（正解） |

### 解釈（正直に）

- ✅ **オンデバイスは成立**。iPad 内だけで手書きを正しく認識（`2x+19=9`）。最終目標＝端末内完結を実機で実証。
- ✅ **8GB iPad に収まる**。ピーク 3.2GB、残 ~1.96GB → 8GB の Pro 最廉価構成でも余裕あり（[ON_DEVICE_FEASIBILITY.md](ON_DEVICE_FEASIBILITY.md) の見立てを実機で裏取り）。
- ⚠️ **速度は Mac/クラウドより遅い**: iPad warm **2.38s** vs Mac(Ollama) 0.23s ＝ **約 10 倍**。要因は画像の vision encoder 処理。瞬時ではないが「タップ→認識」+確認カード UX なら実用域。初回のみモデル DL ~3.2GB。
- ⚠️ サンプルは 1 種（短い一次式）。長い式・多トークンでは latency が伸びる見込み → 今後 latency×式長を計測予定。

### 速度を詰める余地（次の研究）

- 画像解像度を絞る（現状 1024px リサイズ）／プロンプト短縮でプリフィル削減。
- より小さい VLM（SmolVLM, Qwen2-VL 2B）で latency↓ を測り精度とのトレードオフを見る。
- 量子化（4bit→3bit 等）や KV キャッシュ調整。

## 面談での語り方

「ローカル LLM の手書き認識を**自分の実データで定量評価**（初期 5/5）。印刷用 OCR では破綻することも実測。完璧でない前提で**確認＋検算の二層**を入れて実用化した」= 計測に基づく設計判断。
さらに**端末内に載るサイズ（3B/E2B）で横並び比較**し、精度・速度・メモリの実測から **8GB iPad は Qwen2.5-VL 3B が最適**と結論。計測中に見つけたパーサの表記ゆれ弱点も即修正＝「測って・直して・選ぶ」を回した。
